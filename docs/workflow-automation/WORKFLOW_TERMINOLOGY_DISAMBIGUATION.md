# Workflow Terminology Disambiguation

**Issue**: Two distinct meanings for the word "workflow" in the repository  
**Priority**: DOCUMENTATION (prevents contributor confusion)  
**Date**: 2026-01-01  
**Status**: Documented for clarity

---

## 🎯 Executive Summary

The guia_js repository uses the term **"workflow"** in **two completely different contexts**, which can cause confusion:

1. **`.ai_workflow/`** - AI automation logs and artifacts (external tool)
2. **`.github/workflows/`** - GitHub Actions CI/CD configuration (native GitHub)

**Key Distinction**: Same word, different domains, no relationship between them.

---

### 2. GitHub Workflows (`.github/workflows/`)

**Location**: `.github/workflows/` directory  
**Purpose**: CI/CD automation configuration for GitHub Actions  
**Source**: GitHub's native CI/CD platform  
**Nature**: **Configuration files** (YAML definitions)

#### Directory Structure

```
.github/workflows/
├── README.md                           # Technical documentation (7.6KB)
├── copilot-coding-agent.yml            # Primary validation workflow (107 lines)
├── documentation-lint.yml              # Documentation quality checks (156 lines)
└── modified-files.yml                  # Smart change detection workflow (316 lines)
```

**Total**: 3 active workflows, 579 lines of YAML configuration

#### Workflow Purposes

##### 1. `copilot-coding-agent.yml` (Primary Validation)

**Triggers**: Push to main/develop, PRs to main  
**Duration**: ~30-60 seconds  
**Jobs**:

- JavaScript syntax validation (`node -c`)
- Basic functionality test (`node src/guia.js`)
- Web server connectivity test
- Required files verification
- Code style checks
- Security scanning

**Example**:

```yaml
name: Copilot Coding Agent

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
      
      - name: Validate JavaScript syntax
        run: node -c src/guia.js && node -c src/guia_ibge.js
```

##### 2. `documentation-lint.yml` (Documentation Quality)

**Triggers**: .md file changes, PRs, push to main  
**Duration**: ~1-3 minutes  
**Jobs**:

- Markdown linting (markdownlint-cli2)
- Link validation (internal/external)
- Line number deprecation checks
- Badge synchronization
- Version consistency validation

**Example**:

```yaml
name: Documentation Quality Check

on:
  pull_request:
    paths:
      - '**.md'
  push:
    branches:
      - main

jobs:
  markdown-lint:
    name: Markdown Linting
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: DavidAnson/markdownlint-cli2-action@v13
        with:
          globs: '**/*.md'
```

##### 3. `modified-files.yml` (Smart Change Detection)

**Triggers**: Push/PR to main/develop  
**Duration**: ~2-5 minutes  
**Jobs**:

- Change detection (JS, tests, docs, src)
- Affected tests execution
- Test documentation updates
- Documentation validation
- Coverage badge updates
- Workflow summary generation

**Example**:

```yaml
name: Modified Files - Test and Documentation Updates

on:
  pull_request:
    branches: [ main, develop ]
  push:
    branches: [ main, develop ]

jobs:
  detect-changes:
    name: Detect File Changes
    runs-on: ubuntu-latest
    outputs:
      js_files: ${{ steps.filter.outputs.js }}
      test_files: ${{ steps.filter.outputs.tests }}
    steps:
      - name: Detect changed files
        run: |
          git diff --name-only HEAD~1 HEAD
```

#### Characteristics

- **Executable YAML**: GitHub Actions interprets and runs these
- **Triggered Automatically**: On push, PR, schedule, or manual dispatch
- **Cloud Execution**: Runs on GitHub's infrastructure (ubuntu-latest)
- **Version Controlled**: Changes to workflows are tracked in git
- **Developer Modified**: Team actively edits and improves workflows

#### Purpose in Development Lifecycle

GitHub workflows:

1. **Validate** code changes before merge
2. **Run** automated tests on every commit
3. **Check** documentation quality
4. **Update** metadata and badges
5. **Prevent** broken code from reaching main branch
6. **Report** test results and coverage

**Key Point**: This is about **CI/CD automation**, not AI-assisted development.

## 🔍 Key Differences

| Aspect | AI Workflow (`.ai_workflow/`) | GitHub Workflows (`.github/workflows/`) |
|--------|-------------------------------|----------------------------------------|
| **Purpose** | AI automation logs & artifacts | CI/CD configuration |
| **Nature** | Output directory (logs) | Configuration files (YAML) |
| **Source** | External AI tool | GitHub Actions platform |
| **Location** | Repository root | `.github/` subdirectory |
| **Executable** | No (logs only) | Yes (by GitHub Actions) |
| **Created By** | AI workflow tool | Developers |
| **Modified By** | AI tool automatically | Developers manually |
| **Trigger** | Manual AI tool execution | Git events (push, PR, schedule) |
| **Runs On** | Local machine or AI server | GitHub's cloud infrastructure |
| **Version Control** | Optional (can be gitignored) | Mandatory (must be in git) |
| **File Types** | `.log`, `.md` (prompts/summaries) | `.yml` (YAML configuration) |
| **Frequency** | Occasional (AI analysis runs) | Every push/PR |
| **Output** | Logs, prompts, summaries | Test results, validation status |
| **Audience** | Project maintainers | Developers, CI/CD system |
| **Lifespan** | Historical archive | Active configuration |
| **Relationship** | Standalone (no dependencies) | Depends on repository code |

## ⚠️ Potential Confusion Scenarios

### Scenario 1: "Check the workflow logs"

**Ambiguous Request**:
> "Can you check the workflow logs to see what failed?"

**Clarification Needed**:

- **AI Workflow logs**: Check `.ai_workflow/logs/workflow_YYYYMMDD_HHMMSS/`
- **GitHub workflow logs**: Check GitHub Actions tab in repository

**Correct Phrasing**:

- "Check the **AI workflow** logs in `.ai_workflow/`"
- "Check the **GitHub Actions** logs on the Actions tab"

---

### Scenario 2: "Modify the workflow configuration"

**Ambiguous Request**:
> "We need to modify the workflow configuration"

**Clarification Needed**:

- **AI Workflow config**: Edit `.workflow-config.yaml`
- **GitHub workflow config**: Edit `.github/workflows/*.yml`

**Correct Phrasing**:

- "Modify the **AI workflow** configuration (`.workflow-config.yaml`)"
- "Modify the **GitHub Actions** workflow (`.github/workflows/`)"

---

### Scenario 3: "Run the workflow"

**Ambiguous Request**:
> "Let's run the workflow to test this"

**Clarification Needed**:

- **AI Workflow**: Execute the AI automation tool manually
- **GitHub workflow**: Trigger GitHub Actions (push, PR, or manual dispatch)

**Correct Phrasing**:

- "Run the **AI workflow automation tool**"
- "Trigger the **GitHub Actions workflow** (push or manual run)"

---

### Scenario 4: "Workflow failed"

**Ambiguous Report**:
> "The workflow failed yesterday"

**Clarification Needed**:

- **AI Workflow**: Check `.ai_workflow/logs/` for error logs
- **GitHub workflow**: Check GitHub Actions tab for failed runs

**Correct Phrasing**:

- "The **AI workflow execution** failed (check `.ai_workflow/logs/`)"
- "The **GitHub Actions workflow** failed (check Actions tab)"

## 🔄 Workflow Interactions

### Do They Interact?

**Short Answer**: No

### Detailed Explanation

**AI Workflow** (`.ai_workflow/`):

- Runs independently
- Analyzes repository state
- Generates recommendations
- Logs execution details
- **Does not trigger GitHub Actions**
- **Does not depend on GitHub Actions**

**GitHub Workflows** (`.github/workflows/`):

- Triggered by git events
- Validates code changes
- Runs tests automatically
- **Does not read `.ai_workflow/` logs**
- **Does not trigger AI workflow tool**

### Independence Principle

```
AI Workflow Tool                    GitHub Actions
     ↓                                    ↓
.ai_workflow/                     .github/workflows/
     │                                    │
     │                                    │
     └────── No Direct Interaction ──────┘
```

**Both target the same repository, but operate independently.**

## 🎯 Best Practices

### When Discussing Workflows

1. **Always Qualify**:
   - ❌ "Check the workflow"
   - ✅ "Check the **GitHub Actions** workflow"
   - ✅ "Check the **AI workflow** logs"

2. **Use Context Clues**:
   - If discussing CI/CD → GitHub workflows
   - If discussing AI analysis → AI workflow
   - If discussing logs → Specify which logs

3. **Reference File Paths**:
   - "Check `.ai_workflow/logs/`" (clear)
   - "Check the Actions tab on GitHub" (clear)
   - "Check the workflow logs" (ambiguous)

4. **Use Visual Distinctions**:
   - 🤖 AI Workflow (automation tool)
   - 🔄 GitHub Workflow (CI/CD)

## ✅ Conclusion

The term **"workflow"** has **two distinct, unrelated meanings** in this repository:

1. **AI Workflow** (`.ai_workflow/`) - External AI automation tool outputs
2. **GitHub Workflows** (`.github/workflows/`) - Native CI/CD automation

**Key Takeaway**: Always qualify which workflow you're referring to:

- 🤖 "AI workflow" or "AI workflow tool"
- 🔄 "GitHub workflow" or "GitHub Actions workflow"

**Never say**: "The workflow" (too ambiguous)

**When in doubt**: Provide the file path or location to eliminate confusion.
